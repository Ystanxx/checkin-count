# 业务规则冻结

本文件是新系统唯一业务规格基线。若旧 README、旧注释与旧代码冲突，
以旧代码真实行为为审计输入，再以本文件的最终结论为准。

## 1. 数据来源与处理边界

1. 输入文件支持：
   - `.xls`
   - `.xlsx`
   - `.xlsm`
2. 支持多文件、多工作表联合处理。
3. 所有 Excel 读取、解析、聚合、导出、规则判断都在 Rust 侧完成。
4. 前端只负责：
   - 文件选择
   - 规则配置
   - 结果预览
   - 导出触发
   - 运行日志展示
5. 系统默认本地离线运行，不出网。

## 2. 人员块识别规则

### 2.1 旧结构基线

旧系统按“三行一人”理解原表：

1. 第一行：姓名信息
2. 第二行：日期信息
3. 第三行：对应日期的时间 token

### 2.2 新系统识别规则

新系统仍兼容“三行一人”，但不再依赖固定硬门槛，而采用评分式探测：

1. **姓名行识别**
   - 优先识别`姓名：张三`、`姓名:张三`
   - 兼容`姓名 | 张三`这类“姓名标签 + 右侧值”结构
   - 兼容全角字符与空白变体
2. **日期行识别**
   - 识别`1~31`日号
   - 不再要求必须有`>=5`个日期列
   - 以“日期密度 + 连续性 + 与姓名行相对位置”综合评分
3. **时间行识别**
   - 可读取日期行之后的连续一至多行
   - 直到遇到下一个人员块信号、明显的下一条日期行或结束
4. **异常行处理**
   - 工号、备注、空行、部门行不应导致整表失败
   - 单人块解析失败时，只隔离该块，不拖垮整文件
5. **姓名全集**
   - 只要识别出姓名，就必须进入姓名全集
   - 即使没有任何有效打卡，也必须进入后续聚合链路

## 3. 时间 token 规范化

新系统必须支持以下输入格式，并统一转换为强类型时间值：

1. `H:MM[:SS]`
2. `HH:MM[:SS]`
3. `HMM`
4. `HHMM`
5. Excel 时间小数
6. 全角数字、全角冒号、全角分隔符

分隔符至少支持：

- 空格
- 换行
- 逗号
- 分号
- 斜杠
- 顿号
- 全角等价符号

## 4. 打卡窗口规则

### 4.1 规则来源

1. 所有窗口边界必须来自`AttendanceRules`配置。
2. 核心业务代码中不允许散落硬编码窗口常量。

### 4.2 默认预设

根据旧代码`app/processor.py`的真实实现，新系统默认预设为：

- `AM`窗口：`00:00:00 ~ 09:11:59`
- `NOON`窗口：`11:00:00 ~ 14:11:59`

因此在默认预设下：

- `09:11:59`计入有效`AM`
- `09:12:00`不计入`AM`
- `14:05:00`计入有效`NOON`
- `14:12:00`不计入`NOON`

### 4.3 结论

新系统明确冻结：

1. `14:05`在默认规则下**计入有效 NOON**
2. 未来可以通过规则配置切换窗口，但默认口径以上述值为准

## 5. 逐日聚合规则

1. 同一人、同一天、同一窗口，多次打卡只记`1`次。
2. 同一人、同一天，最多记`2`次：
   - `AM 1次`
   - `NOON 1次`
3. 无论原始 token 有多少条，只要窗口命中，窗口计数就是`0/1`。
4. 逐日明细必须保留命中窗口与标准化后的时间列表，便于追溯。

## 6. 需要打卡日规则

1. 用户输入年份、月份、休息日集合。
2. `需要打卡日 = 当月自然日 - 休息日`
3. `应打卡次数 = 需要打卡日 * 2`
4. 需要打卡日表必须可单独导出。

## 7. 汇总指标定义

按单人维度输出以下字段：

1. `姓名`
2. `需要打卡日`
3. `应打卡次数`
4. `打卡天数`
5. `打卡次数`
6. `缺勤天数`
7. `缺勤次数`
8. `缺勤具体日期`

定义如下：

1. `打卡天数`
   - 在需要打卡日范围内，只要当天任一窗口命中，则该日记为`1`天
2. `打卡次数`
   - 在需要打卡日范围内，逐日窗口去重后的总次数
3. `缺勤天数`
   - 只统计`0`次打卡的需要打卡日数量
4. `缺勤次数`
   - `应打卡次数 - 打卡次数`
   - 只打一窗的日期会贡献`1`次缺勤，但不贡献缺勤天数
5. `缺勤具体日期`
   - 仅列出`0`次打卡的需要打卡日

## 8. 0 打卡人员规则

这是新系统强制规则：

1. 只要姓名被识别出来，即使没有任何有效打卡，也必须：
   - 出现在汇总表
   - 参与缺勤天数、缺勤次数计算
   - 参与通报名单判断
2. 不能因为解析器没收集到打卡 token，就把该人从汇总链路中丢掉。

## 9. 通报名单规则

### 9.1 触发条件

支持以下两类阈值：

1. `缺勤天数 > X`
2. `缺勤次数 > X`

支持三种启用方式：

1. 只启用天数阈值
2. 只启用次数阈值
3. 同时启用，并通过`AND/OR`组合

### 9.2 输出字段

通报名单至少包含：

1. `姓名`
2. `需要打卡日`
3. `应打卡次数`
4. `打卡天数`
5. `打卡次数`
6. `缺勤天数`
7. `缺勤次数`
8. `缺勤具体日期`
9. `触发原因`

### 9.3 排序

默认按以下优先级排序：

1. 缺勤天数降序
2. 缺勤次数降序
3. 姓名升序

## 10. 导出规则

系统必须支持：

1. 汇总`xlsx`
2. 汇总`csv`，并带`UTF-8 BOM`
3. 可选明细 sheet
4. 可选需要打卡日 sheet
5. 通报名单 sheet 或独立文件

导出要求：

1. 中文在 Excel 中打开不乱码
2. 文件名默认包含年月与导出类型
3. 导出路径必须经过白名单规范化
4. 前端不直接写文件

## 11. 安全与隐私规则

1. 默认本地处理，不出网
2. 不启用不必要的 shell/HTTP 权限
3. 路径输入只允许来自系统文件选择器或受控保存路径
4. 用户可见错误信息不得泄漏调试堆栈和完整绝对路径
5. 日志默认不打印完整个人数据
